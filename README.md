# Прототип системы детекции уборки столиков по видео

Минимальный рабочий прототип для **одного видео** и **одного столика**.

В этом варианте используется **YOLO (Ultralytics)** для детекции людей и простая логика по фиксированной зоне столика (ROI).

## Что делает прототип

Скрипт:
- детектирует людей на каждом кадре;
- анализирует **только один выбранный столик**;
- определяет 3 ключевых события:
  - `empty` — стол пустой;
  - `occupied` — стол занят;
  - `approach` — к столу подошел человек после периода пустоты;
- записывает события в `events.csv`;
- считает статистику по интервалам `empty -> approach`;
- рисует на видео рамку столика:
  - **зеленый** — пусто;
  - **красный** — занято;
  - **желтый** — переходное состояние во время debounce.

## Используемая логика

1. На каждом кадре YOLO ищет объекты класса `person`.
2. Для каждого найденного человека проверяется, относится ли он к выбранному столику:
   - нижняя центральная точка bbox попадает в ROI столика, **или**
   - достаточная часть bbox пересекается с ROI.
3. Если в зоне столика есть хотя бы один человек, состояние считается `occupied`, иначе `empty`.
4. Чтобы убрать случайные скачки, используется `debounce_sec`: новое состояние должно держаться заданное время, прежде чем будет зафиксировано как событие.
5. Когда после `empty` фиксируется переход в `occupied`, дополнительно создается событие `approach`.

## Структура проекта

- `main.py` — основной скрипт.
- `requirements.txt` — зависимости.
- После запуска формируются:
  - `output.mp4` — видео с визуализацией;
  - `events.csv` — лог событий;
  - `report.txt` — краткий отчет;
  - `problem_frame.jpg` — пример спорного кадра (если такой найден автоматически).

## Установка

```bash
pip install -r requirements.txt
```

## Как запустить

### Вариант 1. Задать ROI вручную

```bash
python main.py --video video1.mp4 --table_roi 530,280,240,160
```

### Вариант 2. Выбрать ROI мышкой через OpenCV

```bash
python main.py --video video1.mp4 --select_roi
```

### Пример полного запуска

```bash
python main.py \
  --video video1.mp4 \
  --table_roi 530,280,240,160 \
  --output output.mp4 \
  --events_csv events.csv \
  --report_txt report.txt \
  --problem_frame problem_frame.jpg
```

## Аргументы

- `--video` — входное видео.
- `--output` — выходное видео.
- `--events_csv` — CSV с событиями.
- `--report_txt` — текстовый отчет.
- `--problem_frame` — файл для проблемного кадра.
- `--table_roi` — координаты столика `x,y,w,h`.
- `--select_roi` — интерактивный выбор ROI.
- `--model` — модель YOLO, по умолчанию `yolov8n.pt`.
- `--conf` — confidence threshold.
- `--device` — устройство (`cpu`, `0` и т. д.).
- `--debounce_sec` — защита от шумовых переключений.
- `--min_ioa` — минимальная доля bbox человека в ROI.
- `--show` — показывать видео в окне.

## Какое видео и какой столик выбраны

В текущем архиве **конкретное видео не включено**, потому что оно не было загружено в среду.

Для итоговой сдачи заполни этот раздел после прогона:
- **Видео:** `video1.mp4`
- **Выбранный столик:** например, столик у правого края кадра / центральный столик / столик у окна
- **ROI:** `x,y,w,h`

## Полученный результат

В этой среде **результат не вычислен**, потому что исходное видео не было предоставлено.

После запуска смотри:
- `report.txt` — там будет среднее время между уходом гостя и подходом следующего человека;
- `events.csv` — там будут все временные метки событий.

## Пример проблемного кадра

`problem_frame.jpg` сохраняется автоматически, если найден спорный кадр, где человек близок к зоне столика, но не дотягивает до порога пересечения. Такой кадр удобно вставить в отчет как пример ограничения метода.

## Ограничения прототипа

- Анализируется только **один** столик.
- Нет разделения на гостя и сотрудника.
- Нет трекинга ID человека между кадрами.
- Возможны ложные срабатывания, если человек проходит очень близко к столику.
- Качество сильно зависит от удачности выбора ROI.
